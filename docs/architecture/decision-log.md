# Architectural Decision Log

This log tracks the most important architectural, philosophical, metaphysical, and experiential decisions behind Liahona.

## Artifact-first homepage

Decision:
The first experience centers on the Liahona artifact, not a dashboard.

Why:
The object is the strongest orientation anchor for the site and should carry the first impression through presence, light, motion, and restraint.

Implication:
Operational tools stay available, but the homepage should remain quiet, immersive, and artifact-led.

## Continuity device as secondary overlay

Decision:
The continuity device is a secondary overlay for logs, sources, return points, fragments, and operational continuity.

Why:
Continuity matters, but it should not compete with the artifact as the primary world.

Implication:
The device should open intentionally, behave predictably, and return cleanly to the Liahona view.

## Fewer features, higher quality interaction

Decision:
The experience should prioritize fewer features with stronger interaction quality.

Why:
Depth, smoothness, reliability, and restraint matter more than visible feature density.

Implication:
New features should be added only when they improve orientation, continuity, or grounded interaction.

## Christ as orientation, not persona

Decision:
Christ may be treated as an orienting center, but not as a simulated persona.

Why:
Liahona is a human-built system and should not present itself as divine, prophetic, authoritative, or revelatory.

Implication:
The system can point toward grounding, scripture, and judgment, but it should not roleplay Christ or speak as Christ.

## Scripture/source grounding, not roleplay

Decision:
Scripture and sources should function as grounding references, not as material for performance or roleplay.

Why:
Grounding improves truthfulness and context while keeping the system restrained and non-authoritative.

Implication:
Source lists and references should be visible, traceable, and clearly framed as grounding material.

## Operational continuity should stay visible but not dominate the experience

Decision:
Logs, versions, decisions, and return points should remain accessible without becoming the first-load experience.

Why:
The project needs recoverable continuity, but visible operations can overwhelm the artifact-first direction.

Implication:
Operational continuity belongs inside the continuity device and supporting documentation.

## Project history should be semantic and recoverable

Decision:
Project history should be organized around meaningful decisions, changes, versions, and return points.

Why:
Git remains the technical rollback system, but people need a readable map of why the project changed.

Implication:
The site and docs should preserve project memory in concise, semantic records that are easy to revisit.
