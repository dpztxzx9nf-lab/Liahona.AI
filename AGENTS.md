# Project Identity

- Liahona.AI is artifact-first.
- The Liahona artifact is the primary experience.
- The continuity device is secondary.
- Christ is orientation, not persona.
- Scripture/source grounding should point outward, not pretend authority.
- Preserve reverence, restraint, and clarity.

# Workflow Rule

Use Diagnose -> Propose -> Wait for uncertain changes.

For uncertain tasks:

1. Inspect files first.
2. Explain the likely cause.
3. Propose the safest fix.
4. List files that would change.
5. Wait for approval before editing/committing/pushing.

# Safe Change Levels

Easy:

- docs
- text
- CSS
- image swaps
- spacing
- animation polish

Moderate:

- HTML structure
- UI state logic
- localStorage behavior
- navigation flow

Careful:

- dependencies
- deployment config
- persistent storage
- routing
- data models

# Git Workflow

- Always run git status first.
- Keep commits concise and semantic.
- Commit only after checks pass.
- Push only when explicitly requested or when the prompt specifically asks for it.
- Create checkpoints before major architectural or UI changes.

# Verification

- Run available checks before commit.
- Summarize manual verification steps for UI work.
- For static asset changes, use versioned filenames when appropriate to avoid cache confusion.

# Boundaries

- Do not over-engineer.
- Do not add major features during polish tasks.
- Preserve artifact-first hierarchy.
- Preserve mobile usability.
- Preserve continuity modal behavior unless explicitly changing it.
- Preserve Decisions and continuity architecture.

# Architectural Direction

Front:

- symbolic encounter
- experiential artifact
- motion/light/orientation
- no text overload

Back:

- scripture grounding
- concise canonical references
- outward Gospel Library links
- sourcefulness over explanation

# Continuity Philosophy

The continuity system preserves meaningful architectural and experiential evolution across time.

Do not memorialize noise.

Prefer meaningful milestones, decisions, and directional transitions.
